# GUIA-Madrid-Bench Efficiency Scripts

This folder contains the scripts used to monitor RDF generation executions and compute efficiency metrics for **GUIA-Madrid-Bench**.

The goal of these scripts is to evaluate not only how much RDF a system generates, but also how efficiently it does so and how much correct information it produces per unit of time.

## What is evaluated?

Efficiency evaluation complements effectiveness evaluation.

While effectiveness metrics compare generated RDF graphs against the gold standard, efficiency metrics analyse the computational and operational cost of producing those RDF graphs.

### Resource consumption metrics

```text
total execution time
CPU usage
RAM consumption
GPU usage, when available
VRAM consumption, when available
energy consumption
CO2 emissions
```

### Generation efficiency metrics

```text
documents processing rate
triples per second
average document latency
```

### Correctness-aware efficiency metrics

```text
valid triples per second
net utility rate
semantic speed index
time-to-target value
```

These metrics are useful because a system may be fast but generate many incorrect triples. Correctness-aware metrics penalize this behaviour by combining time, generated triples, and effectiveness scores.

## Expected workflow

The efficiency evaluation follows this workflow:

1. Run the RDF generation system while collecting monitoring information.
2. Store the generated RDF predictions in a technique-specific folder.
3. Run the effectiveness evaluation, because some efficiency metrics reuse correctness values such as micro precision or semantic similarity.
4. Run `run_efficiency.py` to compute the efficiency metrics.
5. Optionally compare several techniques using the generated CSV reports.

## Expected resources layout

```text
resources/
├── gold/
├── inputs/
│   └── corpus/
├── ontology.ttl
├── shapes.ttl
├── predictions/

└── reports/
    └── my_system/

        ├── effectiveness_strict_content_summary.csv
        ├── effectiveness_fuzzy_content_summary.csv
        └── ...
```

## Prerequisites

Before running `run_efficiency.py`, the generation system must have been executed under the monitoring script, so that the following file exists:

```text
monitor_results_summary.csv
```

This file is required because the efficiency evaluator needs the total execution time:

```text
T_seconds
```

Without this file, time-based metrics cannot be computed.

Some correctness-aware efficiency metrics also require outputs from the effectiveness evaluation, such as strict micro precision, micro F1, or semantic similarity. Therefore, the recommended order is:

```text
1. monitor the generation process
2. run effectiveness evaluation
3. run efficiency evaluation
```

## Running efficiency evaluation

From this folder:

```bash
python run_efficiency.py   --pred-dir ../resources/predictions/my_system   --gold-dir ../resources/gold   --ontology ../resources/ontology.ttl   --output-dir ../resources/reports/my_system
```

Where:

- `--pred-dir` points to the RDF predictions generated by the evaluated system;
- `--gold-dir` points to the RDF gold standard;
- `--ontology` points to the GUIA ontology;
- `--output-dir` points to the folder containing monitoring and effectiveness reports, and where efficiency results will be written.

## Output files

The efficiency evaluator writes:

```text
efficiency_results.csv
```

The monitoring workflow may also generate:

```text
hardware_specs.json
monitor_results.csv
monitor_results_summary.csv
```

## Interpreting results

Efficiency metrics should not be interpreted in isolation.

For example:

- a high triples-per-second value may indicate fast generation, but not necessarily correct generation;
- a high documents-processing-rate value may be misleading if most outputs are invalid RDF;
- valid-triples-per-second is more informative when comparing systems with different output quality;
- net utility rate penalizes systems that generate many false positives;
- semantic speed index relates semantic similarity to execution time.

In practice, efficiency results should be interpreted together with the effectiveness reports generated by `effectiveness_scripts/`.

## Minimal full workflow

```bash
# 1. Generate predictions with your RDF generation system and monitor its execution.
# The output should include RDF files under:
# ../resources/predictions/my_system/
# and monitoring summaries under:
# ../resources/reports/my_system/

# 2. Run effectiveness evaluation.
cd ../effectiveness_scripts

docker compose run --rm guia-metrics python effectiveness.py effectiveness report   --pred-dir /resources/predictions/my_system   --gold-dir /resources/gold   --ontology /resources/ontology.ttl   --shapes /resources/shapes.ttl   --output-dir /resources/reports/my_system

# 3. Run efficiency evaluation.
cd ../efficiency_scripts

python run_efficiency.py   --pred-dir ../resources/predictions/my_system   --gold-dir ../resources/gold   --ontology ../resources/ontology.ttl   --output-dir ../resources/reports/my_system
```

## Notes

- Predictions and gold standards are matched by filename stem.
- Efficiency results depend on the execution environment.
- Hardware information should always be reported when comparing systems.
- GPU and VRAM metrics may be unavailable if no compatible GPU is detected or if the process does not have permission to access GPU telemetry.
- CO2 and energy estimates depend on the monitoring library and the detected execution environment.
